# TDD & Layer 指南（唯一有效版）

> 本文件是 **唯一有效的 TDD 计划**。  
> 如有其它名字相近的文件（例如 `tdd-plans.md`），一律视为历史兼容 / 跳转占位。

---

## 1. Layer 定义

- **T0 – Running**
  - 端到端最小流程能跑通。
  - 允许“先写实现，后补测试”，重点是打通链路。

- **T1 – Stable**
  - 核心路径（网络、同步、AI 解析、导出）有可靠测试。
  - 关注错误处理、重试、超时、边界条件。

- **T2 – Polish**
  - UI/交互改进、动画、布局。
  - 以人工测试为主，可选 UI 测试。

- **T3 – Extra**
  - 实验性 / 附加功能。
  - 是否写测试按性价比决定。

---

## 2. 必须优先测试的“关键路径”

在以下区域，**T1 阶段优先写或补测试**：

1. **连接与同步**
   - BLE 连接状态机；
   - Wi-Fi 凭据传输；
   - 设备与手机之间的重试 / 超时逻辑。

2. **AI & 文本处理**
   - 调用 Qwen / 其它模型的封装层；
   - Tingwu / 语音转写结果解析；
   - 关键业务 prompt 组装逻辑。

3. **导出 / 数据落地**
   - Markdown → PDF/CSV 等导出；
   - 本地草稿 / 缓存保存与恢复。

> 对这些区域：新增功能时优先考虑“先写测试，再写实现”；  
> 已有功能则尽量补上覆盖核心 happy path + 若干关键错误路径。

---

## 3. 可以先实现后补测试的场景

- 纯 UI 原型 / 布局探索（T0/T2）；  
- 仍在探索中的接口 / 不确定是否保留的功能；  
- 非关键路径的小工具函数。

但建议在功能稳定后：

- 补上一些基本测试，至少覆盖主要 happy path；  
- 在 `docs/progress-log.md` 中说明“已补测”或“仍缺测试”的部分。

---

## 4. 与其他文档的关系

- `dev_plan.md`：决定当前在哪个模块 & Layer 上工作；  
- 本文件：说明在该 Layer 上测试应扮演的角色；  
- `docs/progress-log.md`：记录每次实际是否写了测试，以及测试范围。

> 记忆口诀：  
> - T0：先跑起来；  
> - T1：用测试拧紧；  
> - T2/T3：看性价比安排测试。
