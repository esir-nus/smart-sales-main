## Orchestrator–MetaHub V1–V7 演进小结（股东视角）

> 这是一份给股东 / 业务方看的“故事版”说明，回答三个问题：  
> 1）系统这几年到底升级了什么？  
> 2）为什么 V7 是一个重要拐点？  
> 3）这对业务和投资回报有什么实际影响？
>
> 说明：V7 已归档；当前规范为 `docs/Orchestrator-V1.md`。本文仅用于历史叙事。

---

### 一、从 V1 到 V7：一句话版本对比

下表只看“整体感觉”和“能不能托付长音频资产”，刻意弱化技术细节：

| 版本 | 核心特性（一句话） | 用户体验（尤其长音频） |
|------|--------------------|-------------------------|
| **V1** | 能调用大模型给出答案 | 适合做短问题 Demo，长音频不稳定、难复现，更多是“玩具级体验” |
| **V2** | 加了一层中间控制 | 一次录音大多能跑完，但一旦出错，需要工程师翻大量日志才能定位问题 |
| **V3** | 开始区分“给人看”和“给系统用”的内容 | 长一点的录音可以跑，但出问题时很难还原“系统当时到底做了什么”，不利于排查和复盘 |
| **V4** | 正式把“聊天内容”和“机器可读 JSON”分开 | 聊天气泡变干净了，但长音频仍然是“整段跑完再看结果”，中途几乎没有可视反馈 |
| **V5** | 针对 XFyun 链路加上明确护栏和调试信息 | 可以比较稳定地跑几十分钟录音，HUD 里也能看到关键信息，但用户/测试往往要“等全部结束”才能判断好坏 |
| **V6** | 为长音频设计的“批次式智能助手” | 针对一小时级录音：不再一口气等到底，而是按批次逐步产出结果；HUD 里有清晰“进展和证据”，出问题时更容易找到是哪一段出错 |
| **V7** | 在 V6 之上“补齐大脑和仪表盘”的版本 | 默认走更稳的 Tingwu+OSS 通道，把“客户信息+对话结论+会话管理+公司知识”拆成 M1–M4 四层，两类 Agent 分工做分析和润色，HUD 固定为三块可复制的调试面板，让长音频转写和智能分析更稳定、更容易排查问题，也更符合风控与合规要求 |

可以简单理解为：
- V1–V3：证明“能跑起来”，但不适合托管真正的业务；
- V4–V5：开始有协议和护栏，边界更清楚，但体验仍偏工程向；
|- V6：长音频终于“能放心用”，不再一口气等到最后一秒才知道成败；
- V7：在 V6 之上，把“稳定性、可追责性、可扩展性”拉到了可以长期托付业务的水平。

---

### 二、为什么要一路走到 V7？

从股东 / 业务方的角度，我们其实只关心三件事：

1. **稳定可靠**：长音频任务要“十次有九次成功”，而不是“看运气”。  
2. **资产可复用**：沉淀下来的转写和分析结果，将来要能被二次利用，而不是锁死在某一代产品里。  
3. **可审计、可合规**：当销售结论、风控判断有争议时，系统能拿出清晰的“证据链”，同时又不会泄露敏感信息。  

V7 的设计，就是围绕这三点，把前几代的经验教训系统性固化下来。

### 三、各版本的关键变化（简化版）

#### 1）V1–V3：验证“AI 能不能帮忙”，但还不适合托管业务

- 优点：证明了“AI 能听懂录音、能给出结论”，为后续投资打下基础。  
- 问题：  
  - 结果不稳定，同一段录音多次跑，结论可能不一样；  
  - 出问题时，只能靠工程师看一堆日志猜原因；  
  - 聊天气泡里经常混杂调试信息，看起来“不专业”。  

#### 2）V4–V5：把“给人看”和“给机器看”分开，开始有护栏

- 做了什么：  
  - 把“给用户看的内容”和“给系统存档的 JSON”正式分离；  
  - 给 XFyun 这种第三方能力加上“只允许安全模式（transfer-only）”的硬性规则。  
- 带来的好处：  
  - 聊天气泡更干净，调试信息集中到 HUD 里；  
  - 第三方能力“不允许乱试”，减少了“试试看就把配额和口碑赌进去”的风险。  
- 仍然存在：  
  - 长音频体验依然是“一口气等到最后”，过程感不强；  
  - 结构化程度有限，后续想做更智能的分析比较吃力。  

#### 3）V6：让一小时级录音“用起来像一个产品，而不是 Demo”

- 核心改动：  
  - 把长录音切成多个“小批次”，一批一批处理并更新界面；  
  - 引入更清晰的内部结构（例如分段、可疑边界、决策记录等），让系统每一步都有“可追踪的理由”；  
  - HUD 可以看到当前在处理哪一段、做了哪些判断。  
- 对业务的直接意义：  
  - 销售 / 测试不用干等到最后一秒，长会录可以一边跑一边看结果；  
  - 真出问题时，不用整个任务重跑，可以精准定位到“哪一段有问题”。  

#### 4）V7：在 V6 的基础上，补齐“资产分类账本”和“专业仪表盘”

V7 的改动可以分三块理解：

- **更稳的“进水口”**：  
  - 默认统一走 Tingwu+OSS 这条已经验证稳定的转写链路；  
  - XFyun 只在“明确开启 + 能力确认”后才使用，彻底杜绝“先试试看再说”的做法；  
  - 带来的结果是：长音频任务成功率更高、不可预期的失败更少、云端费用更可控。  

- **更清晰的“资产分类账本”（M1–M4）**：  
  - M1：客户自身信息（类似“客户档案”）；  
  - M2：这次对话里推断出的信号（风险、进展、关键句等）；  
  - M3：会话管理信息（会话标题、导出文件名、导出是否放行等）；  
  - M4：公司级知识与话术包（规范用语、标准话术）。  
  - 每一层都有清楚的“谁能写、谁能读、有没有证据”的规则。  
  - 这意味着：以后要做复盘、风控、合规审计或迁移到新系统时，有一套清楚的“总账 + 明细”可以查，而不是被淹没在一坨很难复用的大 JSON 里。  

- **更专业的“仪表盘 + 证据链”（HUD 三块 + 两类 Agent 分工）**：  
  - HUD 固定为三块：  
    1）本次运行的关键设置和状态；  
    2）第三方转写服务返回的原始结果；  
    3）系统预处理后的摘要和可疑边界。  
  - 所有内容都是“可复制、可审计”的，但不会把敏感原始报文直接摊开给普通用户看；  
  - 内部用两类 Agent 分工：  
    - Agent1 负责“记笔记 + 做分析结论”；  
    - Agent2 只负责“把逐字稿润色到专业可读”，不改事实。  
  - 这保证了：当销售 / 客服的智能分析结果有争议时，我们既能给出清晰的证据链，又能向外界说明“系统没有私自篡改原始事实”。

### 四、V7 给股东 / 业务方带来的实际收益

可以用四句话概括 V7 带来的价值：

1. **更稳**：长音频任务“十次有九次能顺利跑完”，异常有清晰的原因和定位方式。  
2. **更省**：不再“乱试第三方能力”，云端费用更可控，失败重跑的浪费更少。  
3. **更值钱**：沉淀下来的转写、分析和会话管理信息，被清晰地分层管理，未来可以在更多产品线上复用，而不是一次性消耗。  
4. **更放心**：当涉及销售结论、合规风控时，系统可以拿出有条理的证据链，同时又不会泄露敏感内容，符合对外宣传的专业形象。  

从投资角度看：  
V6 让 SmartSales 从“AI Demo”变成“可上线的产品”；  
V7 则让这套系统更接近“可以托管长音频资产和销售洞察的基础设施”，为后续在更多业务线、大客户场景中复制提供了底座。
