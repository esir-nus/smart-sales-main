## 阶段三：实现 (IMPLEMENT)

**目标**: 按整洁架构构建。

### 3.1 Box 创建顺序

```
1. Interface → 定义契约（做什么）
2. Fake → 测试替身（供消费者测试）
3. Impl → 真实实现
4. Hilt → @Binds 或 @Provides
5. Tests → 用 Fake 测试
```

### 3.2 基于证据编码

写代码前：
```bash
# 检查是否有类似代码
grep -rn "FeatureName" app-core/

# 检查要遵循的模式
view_file docs/plans/tracker.md
```

### 3.3 Prism 清洁环境规则

> **Prism 代码禁止导入遗留代码。**

| ✅ domain/ 允许 | ❌ domain/ 禁止 |
|-----------------|-----------------|
| `kotlinx.*` | `android.*` |
| `javax.inject.*` | `feature.*` (遗留) |
| `dagger.hilt.*` | `com.smartsales.legacy.*` |

**检查点**: Interface + Fake + Impl + Hilt 绑定完成。

## 🏗️ 交互图谱 (TAR 模型)

每个元素定义遵循此严格逻辑：
- **视觉状态 (Visual)**: 用户交互*前*看到什么。
- **触发器 (Trigger)**: 确切的用户输入 (事件)。
- **动画 (Animation)**: 物理效果或过渡曲线 (反馈)。
- **结果 (Result)**: 逻辑状态变更或导航 (结果)。
- **不变量 (Invariant)**: 绝不可打破的硬性规则 (护栏)。
- **状态 (Status)**: 实现状态 (`✅ 已验证`, `🚧 进行中`, `❌ 待定`)。

### 6.2 Agent 活动横幅 (双层结构)

> **设计原则**: Agent 活动使用 **双层层级** 来丰富展示 Agent 的思考。这是核心差异化点。

> **位置**: AgentActivityBanner 渲染 **在聊天记录中内联** 并作为 Agent 认知的永久记录 **持久化**。当 Agent 完成思考，回复气泡作为 **独立、后续的消息** 出现（而不是替换）。这创造了可见的轨迹：用户 → 思考框 → AI 回复。

#### 视觉结构

```
┌─────────────────────────────────────────────────────────┐
│  Phase: 📝 规划分析步骤                                  │  ← 主层 (高层任务)
│  Action: 🧠 思考中...                                    │  ← 次层 (具体操作)
├─────────────────────────────────────────────────────────┤
│  [思考轨迹 / 转写 / 记忆命中...]                          │  ← 主体 (流式内容)
└─────────────────────────────────────────────────────────┘
```

#### 阶段类型 (Phase)

| 阶段 | 图标 | 文案 | 时机 |
|------|------|------|------|
| `PLANNING` | 📝 | "规划分析步骤" / "生成执行计划" | 分析师模式计划生成 |
| `EXECUTING` | ⚙️ | "执行工具: {toolName}" | 运行工具 (图表, 报告等) |
| `RESPONDING` | 💬 | "生成回复" | 标准回复生成 |
| `ERROR` | ⚠️ | "发生错误" | 错误状态 |

#### 动作类型 (Action)

| 动作 | 图标 | 文案 | 轨迹来源 |
|------|------|------|----------|
| `THINKING` | 🧠 | "思考中..." | Qwen3-Max 原生 CoT 流 |
| `PARSING` | 📄 | "解析中... ({filename})" | Qwen-VL 视觉输出 / OCR |
| `TRANSCRIBING` | 🎙️ | "转写中... ({filename})" | 听悟转写流 |
| `RETRIEVING` | 📚 | "检索记忆..." | 关联库命中 |
| `ASSEMBLING` | 📋 | "整理上下文..." | 组装上下文预览 |
| `STREAMING` | ✨ | "生成中..." | LLM 回复流 |

#### 轨迹来源 (原生 vs 合成)

| 来源 | 类型 | 详情 |
|------|------|------|
| **Qwen3-Max CoT** | 原生 | `enable_thinking=true` → `reasoning_content` (分析师: 完整) |
| **Qwen-Plus CoT** | 原生 | `enable_thinking=true` → `reasoning_content` (教练: 截断3行) |
| **Qwen-VL CoT** | 原生 | `enable_thinking=true` → `reasoning_content` (视觉: OCR 轨迹) |
| **听悟** | 原生 | 实时转写 (伪思考) |
| **关联库** | 合成 | 显示命中的实体/记忆 |
| **上下文组装** | 合成 | 显示组装的来源 |

#### 组件签名

```kotlin
data class AgentActivity(
    val phase: ActivityPhase,        // 必需: PLANNING, EXECUTING, 等
    val action: ActivityAction?,     // 可选: THINKING, PARSING, 等
    val trace: String? = null        // 可选: 流式内容
)

@Composable
fun AgentActivityBanner(
    phase: ActivityPhase,            // 必需: 总是可见
    action: ActivityAction?,         // 可选: 如存在则可见
    trace: List<String>?,            // 可选: 主体内容
    modifier: Modifier = Modifier
)
```

#### 渲染规则

| 场景 | 视觉输出 |
|------|----------|
| `action = null, trace = null` | 仅阶段 (如 "⚠️ 网络连接失败") |
| `action = THINKING, trace = null` | 阶段 + 动作, 无主体 |
| `action = THINKING, trace = [...]` | 阶段 + 动作 + 可滚动轨迹 |

## §4 系统通知 (Snackbar)

> **理念**: 透明的系统状态更新，不阻断用户流程。
> **行为**: 底部出现（在键盘/导航栏上方），4秒后自动消失。

| 通知类型 | 触发条件 | 内容格式 |
|----------|----------|----------|
| **需要澄清** | 实体有歧义 (低置信度) | `⚠️ 需要澄清: [Entity Name] (歧义请求)` |
| **发现冲突** | 检测到日程重叠 | `⚠️ 发现冲突: [Event A] vs [Event B]` |
| **发现关联** | 后台资料库命中 | `✨ 发现关联: [Entity Name] (置信度: [Score]%)` |
| **客户信息更新** | 通过关联库丰富画像 | `👤 客户信息更新: [Name] (新增偏好/职位...)` |
| **知识库扩充** | 从上下文中学习新事实 | `📚 知识库扩充: [Topic] (来源: [Source])` |
| **关系变动** | 检测到实体连接 | `🔗 关系网变动: [Person A] -> [Person B]` |
| **记忆刷新** | 用户画像/上下文更新 | `🧠 记忆已更新: [Key Information]` |
| **听悟完成** | 后台转写完成 | `📝 转写完成: [File Name] (点击查看)` |
| **任务已创建** | 语音指令处理完毕 | `✅ 日程已创建: [Task Name] [Time]` |

**视觉风格:**
*   **冲突/警告**: 琥珀色/黄色强调。
*   **关联/信息**: Prism 蓝/紫色强调。
*   **动作**: 点击 Snackbar 跳转到相关卡片或上下文。

## 1. 执行摘要 (Executive Summary)

**Prism** 是智能个人助理系统的架构标准。它用一个 **统一管道 (Unified Pipeline)** 取代了复杂的状态协调，并将具体行为委托给 **模式策略 (Mode Strategies)**。

"Prism" (三棱镜) 这个名字反映了核心模式：单一的用户意图流进入系统，被 Orchestrator (编排器) "折射" 到特定的执行路径 (Chat, Analyze, Schedule) 中，同时共享 Context (上下文)、Memory (记忆) 和 Intelligence (智能) 的通用基础。

### 核心原则
1.  **统一管道**: 所有模式共享相同的高级流程 (Context → Execute → Publish → Persist)。
2.  **策略模式**: 模式特定的逻辑 (LLM 选择, 渲染, 工具) 封装在可切换的策略中。
3.  **即发即弃 (Fire-and-Forget) 持久化**: 记忆写入在后台发生。UI **从不** 阻塞等待数据库操作。
4.  **流式优先**: 架构围绕原生流式传输 (DashScope Flow<Token>) 构建，以实现最小延迟。

## 1. 核心组件 (Core Components)

### 1.1 Context Builder (共享)
智能的基础。无论何种模式，它都会运行。
**角色**: 将异构输入标准化为 Executor LLM 的统一上下文负载。

**输入标准化工具**:
- **Text**: 直通。
- **Audio**: `tingwu.transcribe` → 脚本 + 元数据。
- **Image**: `qwen_vl.analyze` → OCR / 场景描述。
- **URL**: `fetch.text` → 网页内容。

> **架构说明**: Tingwu 从独立工作流降级为**工具**。所有模态现在都是统一的输入源。

**Context Payload (`EnhancedContext`)**:
包含用户文本、音频脚本、图像分析结果、Hot Zone 命中、Relevancy Library 实体上下文、模式元数据。

#### 1.1b Session Cache (任务内记忆)
轻量级内存缓存，用于任务执行期间的快速上下文访问。
- **写入模式**: Tool Output → Session Cache (同步) → Relevancy Library (异步 Delta)。
- **一致性**: 任务执行期间 Session Cache 是真理源。

### 1.2 Orchestrator (The Prism)
无状态路由器。选择模式并调用管道。**绝不**在未经用户确认的情况下自动切换模式。

### 1.3 Executor (基于策略)
大脑。实现随模式变化 (`LlmProvider`)。
- **Chat**: Qwen Turbo (快, 互动)。
- **Analyze**: Qwen Max (深, 推理)。
- **Schedule**: Structured Output LLM (JSON 正确性)。

### 1.4 Publisher (基于策略)
输出整形器 (`ModePublisher`)。
- **ChatPublisher**: 直接流式传输到气泡。
- **AnalystPublisher**: 累积 Token，格式化为章节/洞察，渲染图表卡片。
- **SchedulePublisher**: 解析 JSON，调用日历 API，渲染智能提示。

### 1.5 Memory Writer (后台)
即发即弃 (Fire-and-Forget) + 重试机制。UI **从不** 等待记忆写入。

### 1.6 Schema Linter (硬性质量门禁)
在持久化之前验证所有 AI 生成的结构化输出。**无 LLM 参与验证** — 纯确定性。
验证 Entity, Plan, Scheduler JSON 等。失败则让 Agent 重试 (最多3次)。

### 1.7 Memory Center Notifier
通过 Snackbar 通知用户重要的后台记忆更新 ("习惯已更新: xxxx")。

## 2. 数据流与一致性

### 2.1 缓冲流式传输 (Buffered Streaming)
**权衡**: 我们不能使用真正的原生流式传输，因为 Linter 必须在向用户显示之前验证结构化输出。
**解决方案**: **缓冲流式传输 + 模拟动画**。

```
Orchestrator ──▶ DashScope API (stream=true)
                       ▼
               [Response Buffer] ──accumulate──▶ Complete
                       ▼
               [Linter] ──validate──▶ Pass/Fail?
                       │
         ┌─────────────┴─────────────┐
         ▼                           ▼
 [Publisher] (Simulate Stream)     [Retry]
```

### 2.2 一致性模型
- **UI 响应性**: **Evaluate-Now** (立即显示)。
- **记忆持久化**: **Eventual** (最终一致)。
- **跨会话**: **Strong** (强一致性)。

2.  **Planning**: Planner LLM 输出结构化 `ExecutionPlan`。
3.  **Execute & Respond**: 执行计划，生成带有原生结构化标签的输出。

```
┌─────────────────────────────────────────────────────────────────┐
│                    PRISM MEMORY TRINITY                         │
├─────────────────────────────────────────────────────────────────┤
│                                                                 │
│   ┌─────────────────┐    ┌─────────────────┐                   │
```

| 存储 | 角色 | 内容 |
|------|------|------|
| **Hot Zone** | 活跃工作 | 今日条目，进行中的会话 (`isArchived=false`) |
| **Cement Zone** | 归档 | 完成超过 14 天的条目 (`isArchived=true`) |
| **Relevancy Library** | 客户索引 | 聚合的实体、别名、决策历史 |
| **User Profile** | 静态设置 | 经验水平、行业、角色 |
| **User Habit** | 习得模式 | 会议时间偏好、客户语调等 |

> **注意**: 旧的 "M1/M2/M3/M4" 术语已**在大脑中移除**。请使用上述直观名称。

---

## 2. 双区模型 (Two-Zone Model)

**Hot Zone 定义**:
- 所有 **活跃条目** (`isArchived = false`)。
- 所有 14 天内 (过去或未来) 的 **日程项**。

**Cement Zone 定义**:
- `isArchived = true` 且日期 > 14 天前。

> **机制**: 相同 Schema，不同标志位。条目通过后台压缩从 Hot 老化进入 Cement。

---

## 3. Relevancy Library (关联库)

持久化、直写的索引，用于 O(1) 实体查找和冲突检测。

### 3.1 Schema
```kotlin
data class RelevancyEntry(
    val entityId: String,           // "z-001" (Person)
    val entityType: String,         // PERSON, PRODUCT, LOCATION, EVENT
    val displayName: String,        // "张伟"
    val aliasesJson: String,        // ["张总", "老张"]
    val decisionLogJson: String,    // 决策历史 (用于 Rethink 学习)
    val metricsHistoryJson: String  // 自定义指标的时间序列
)
```

### 3.2 Metrics History (数据可视化)
存储关键销售指标的历史变更。
- **类型**: 货币 (分), 数量, 时长, 枚举, 日期。
- **用途**: Analyst 模式可直接根据此数据生成趋势图 ("显示张总过去6个月的预算趋势")。

### 3.3 字段更新策略
- **displayName**: 首次写入为准。
- **aliasesJson**: 追加 (去重)。
- **metricsHistoryJson**: **追加** (时间序列)。
- **decisionLogJson**: **追加** (历史)。

---

## 4. 实体消歧 (Entity Disambiguation)

当语音输入包含歧义引用 (如 "张总" 匹配多人) 时，系统使用 **基于计数的强化学习**。

### 4.1 别名映射
```kotlin
data class AliasMapping(
    val alias: String,              // "张总"
    val entityId: String,           // "z-001"
    val confirmationCount: Int,     // 确认次数 (强化信号)
    val lastConfirmedAt: Long       // 最近时间
)
```

### 4.2 评分算法
1.  **confirmationCount** (40%): 确认次数。
2.  **recencyDecay** (30%): 时间衰减。
3.  **contextMatch** (30%): 当前会话上下文匹配度。

如果 Top1 得分 > 0.85 且显著高于 Top2 → **自动消歧** (带 Disclosure)。
否则 → **显示选择器**。

---

## 5. 结构化输出模型

LLM 输出分为两个不同部分 — 无内联标记混合。

```
[STRUCTURED SECTION] (JSON)
{ "entities": [...] }

[USER CONTENT] (Clean Text)
「张总希望在周三...」
```

**处理流程**: LLM 输出 → Linter 验证 JSON → Publisher 显示文本 → Writer 写入 JSON 到库。

---

## 6. 用户画像 (User Profile)

用户级配置 (`UserCenter`)。
- **字段**: 经验水平, 行业, 角色, 语言偏好。
- **用途**: 调整 Coaching 深度和建议的专业度。

---

## 7. 用户习惯 (User Habit)

自动学习的行为模式。

**学习原则**:
1.  **观察阈值**: 3次一致观察后创建。
2.  **信心衰减**: 30天无观察则衰减。
3.  **显式优先**: 用户手动设置的永远不会衰减。
4.  **负向信号**: 用户拒绝建议 → 信心大幅下降。

**示例**: 
- `meeting_time`: `morning` (早上)
- `verbosity`: `bullet_points` (要点式)

## 1. 教练模式 (Chat Mode / The Coach)

轻量级对话路径，带有可选的记忆搜索。

### 1.1 流程图 (ASCII Visualization)
```
User Input
    │
    ▼
[Context Builder: Session History]
    │
    ├─▶ LLM Check A: Suggest Analyst? ──(Yes)──▶ Prompt User
    │
    ├─▶ LLM Check B: Memory Threshold Score
    │       │
    │       ├─▶ Score > T: Search Relevancy Library ("Searching memory...")
    │       │
    │       └─▶ Score ≤ T: Session Context Only
    │
    ▼
[Qwen Turbo] ──stream──▶ [ChatPublisher] ──▶ UI
```

### 1.2 上下文策略
1.  **第一步**: 会话历史 (Session History) 为主上下文。
2.  **第二步**: 提示词检查 (Prompt Checks):
    - **Analyst 建议**: 检测是否需要深度分析。需用户确认。
    - **记忆阈值**: 检测是否需要历史上下文 (0-1 分数)。如果 > 阈值，自动搜索 Relevancy Library。

## 2. 分析师模式 (Analyze Mode / The Analyst)

重型推理路径，包含思维轨迹 (Thinking Trace) 和规划表 (Planner Table)。

### 2.1 流程图
```
User Input
    │
    ▼
[Context Builder: Relevancy + Hot Zone]
    │
    ▼
[Qwen Max (Thinking)] ──stream──▶ [Thinking Trace UI]
    │
    ▼
[Qwen Max (Structured)] ──stream──▶ [Planner Table]
    │                                  │
    │                            [Tool Execution]
    │                                  │
    └─async─▶ [Memory Writer] ──▶ Hot Zone
```

### 2.2 上下文策略
1.  查询 **Relevancy Library** (实体上下文, 决策历史)。
2.  查询 **Hot Zone** (最近相关条目)。
3.  生成 `ExecutionPlan` (Plan-Once 模型)。

## 3. 日程模式 (Schedule Mode / Global Top Drawer)

结构化指令路径，具有多步上下文。

### 3.1 流程图
```
Voice Note (ESP32)
    │
    ▼
[Context Builder]
    ├─▶ Step 1: Check Inspirations + Tasks (Today)
    ├─▶ Step 2: Query Relevancy Library (Entity Match)
    └─▶ Step 3: Apply 14-day Rule (Hot vs Cement)
    │
    ▼
[Qwen Structured] ──json──▶ [SchedulePublisher]
                                  │
                            [Calendar API]
                            [Tips Renderer]
```

### 3.2 14天规则 (14-Day Rule)
- 如果 `lastMentionedAt` < 14 天 → 检查 Hot Zone。
- 如果 `lastMentionedAt` ≥ 14 天 → 检查 Cement Zone。

## 4. 模型路由 (Model Router)

基于 **任务类型** 而非模式进行路由。

| 输入条件 | 模型 | 用途 |
|----------|------|------|
| 图片/视频 | `qwen-vl-plus` | 视觉分析 |
| 工具调用 | `qwen3-max` | 32k Tokens |
| 默认 (对话) | `qwen-plus` | 1M Tokens |

**记忆层路由**:
- **RELEVANCY**: `qwen3-max` (精确工具调用)。
- **HOT**: `qwen-plus` (快速导航)。
- **CEMENT**: `qwen-long` (10M Tokens 深层检索)。

---

## 5. 冲突解决 (Rethink Model)

冲突被视为 **创造性的提示 (Creative Prompts)**，而不是简单的合并决策。

**流程**: 检测 → 建议 → 解决

1.  **检测**: 下午 2:00 重叠。
2.  **建议 (Rethink)**: Agent 提出 "第三种选择" (例如：合并为联合会议)。
3.  **用户选择**:
    - **Override**: 覆盖冲突项。
    - **Merge**: 合并两者。
    - **Rethink**: 用户输入自定义解决方案 → 记录到决策日志 (Decision Log)。

## 1. 崩溃恢复与检查点 (Crash Recovery)

**哲学**: 最小检查点策略。只缓存 `EnhancedContext` (昂贵的构建产物)。其他要么已持久化 (聊天记录)，要么构建成本低。

### 1.1 检查点策略
```
CKPT 0: 用户输入 (ChatHistory)
   ↓
CKPT 1: Enhanced Context (ContextCache)
   ↓
CKPT 2: AI 响应 (ChatHistory)
```

**恢复行为**:
- **崩溃于 CKPT 1 之前**: 从头重建上下文 (昂贵)。
- **崩溃于 CKPT 1 之后**: 加载缓存的上下文，恢复 LLM 调用 (跳过昂贵的上下文构建)。

**Context Cache 卫生**:
- **TTL**: 24小时。
- **Scope**: 每个活跃轮次一个检查点。

## 2. 可测试性合约 (Lattice Compliance)

Prism 继承并扩展了 **Lattice 架构** 原则。
- **Prism**: 关注智能 Agent 模式 (记忆, RAG, 流式)。
- **Lattice**: 关注模块化和黑盒隔离。

### 2.1 必需接口与 Fake
每个组件 (Box) 必须有一个接口和一个用于测试的 Fake。

| 接口 | Fake | 用途 |
|------|------|------|
| `LlmProvider` | `FakeLlmProvider` | 返回预设响应 |
| `ContextCache` | `FakeContextCache` | 内存 Map |
| `MemoryRepository` | `FakeMemoryRepository` | 内存存储 |

### 2.2 依赖规则
1.  **禁止 Box-to-Box 导入**: 所有通信必须通过 Orchestrator。
2.  **Domain 层无 Android**: 纯 Kotlin (KMP-ready)。
3.  **Orchestrator 是很薄的**: 委托给 Box，无内联业务逻辑。
