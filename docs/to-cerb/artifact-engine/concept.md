# System IV Concept: The Artifact Engine ⚗️

> **Drafting Protocol**: This document strictly defines the mental model, user experience, and architectural boundaries of the Artifact system. Premature concrete interfaces (`interface.md`) and database schemas are intentionally deferred until the Core Pipeline Test Overhaul is complete.

---

## 1. The Mental Model: What is an "Artifact"?

In the Smart Sales ecosystem, we must distinguish between *Memory* and *Artifacts*:

| Characteristic | SSD Memory (System II) | Artifacts (System IV) |
|---|---|---|
| **Nature** | Raw facts, CRM deltas, user habits. | Synthesized, finalized standalone documents. |
| **Metaphor** | The Agent's internal diary and rolodex. | A printed report or a drafted email sitting on a desk. |
| **Format** | Strict Data Classes / JSON. | Rich Markdown intended for human consumption. |
| **Lifecycle** | Constantly mutating and appending. | Immutable once generated (or overwritten completely). |

An **Artifact** is a distinct, tangible end-product generated by a Plugin (like the Executive Report). It has a unique identity (`AssetId`), a title, a brief context-aware description (so the Agent knows *why* it was created), and a payload of Rich Markdown.

> **Tech Debt Note (Future Extension)**: In the future, this system will expand so when the user uploads external files/docs, the system parses them and registers them as Artifacts with an `AssetId` and short summary, making them queryable by the Sniffer.

## 2. The User Flow & Experience

How does Frank interact with an Executive Report Artifact?

### Stage A: The Factory (Creation)
1. **Trigger**: Frank says, "Generate an executive summary of the Huawei account for exactly what happened this quarter."
2. **Execution**: The `LightningRouter` identifies this as a heavy generative task and routes it to the `ExecutiveReportPlugin` (System III).
3. **Generation**: The Plugin spins up the LLM, synthesizes the CRM history, and produces a comprehensive Markdown report.
4. **Minting**: The Plugin saves this Markdown to the SSD, minting a new Artifact with a unique `AssetId` **and a short descriptive summary** (so the Agent immediately understands the background context of the asset).
5. **Delivery**: The Plugin returns a conversational reply: *"I've generated the Huawei Q3 Executive Report. I've placed it in your Artifact Box."*

### Stage B: The Artifact Box (Discovery & Retrieval)
1. **The Scope**: The **Artifact Box** (the chemical bottle ⚗️ icon) is strictly **session-bounded**. It does not bleed across conversation sessions. It only illuminates when relevant to the *current* context.
2. **Contextual Sniffing**: The user says, *"Huawei is gonna have another round of meetings next week."*
3. **Fast Mapping**: The `ArtifactSniffer` inside ContextBuilder silently matches "Huawei" against the SSD Artifact Library and retrieves the `AssetId` and its short summary.
4. **UI State Update**: The UI state dynamically populates the Artifact Box for this session, listing the historical Huawei Report as available.
5. **The Agent's Response**: Armed with the injected Sniffer context, the LLM organically responds: *"Here's the Huawei report you may find useful, you may check the Artifact Box now."*

### Stage C: The Preview Card (Interaction)
1. **The Click**: Frank taps the Huawei Artifact in the Artifact Box or chat feed.
2. **The Modal**: A Compose Scrim + Drawer (or full-screen Modal) slides up.
3. **The Render**: The raw Markdown payload is rendered into beautiful, native, pro-max UI typography. (No PDFs are generated yet; this is purely visual Markdown rendering).
4. **The Action (Share)**: At the bottom of the Card is a distinct "Share as PDF" action button.
5. **The Export**: Tapping Share triggers Android's native OS Share Sheet (`Intent.ACTION_SEND`). The platform layer bridges the Markdown to a PDF byte stream *at the moment of sharing*, keeping the Domain ignorant of file formats.

## 3. System Boundaries & Cross-Module Connections

When we inevitably write the code, the blast radius will touch these boundaries:

### A. The Storage Layer (`:data:artifact`)
- Needs a new dedicated room database or directory purely for saving Markdown strings attached to UUIDs and semantic descriptions. 
- *Constraint*: Must not mingle with `MemoryRepository` to prevent token poisoning.

### B. The Awareness Layer (`:core:context`)
- Needs an `ArtifactSniffer`. Like the `Alias` dictionary, it must be fast. It maps natural language cues to historical `AssetId` descriptions so the LLM knows what resources it "holds" in its hands.
- *Risk*: Token bloating. Injecting the *full markdown* of an artifact into the context window will crash the LLM. The Sniffer must only inject the `AssetId` + `Title` + `Summary`.

### C. The Presentation Layer (`:app-prism`)
- Needs a robust Markdown-to-Compose rendering engine for the Preview Modal.
- Needs to manage the "Artifact Box ⚗️" state in the `AgentViewModel`.

### D. The Platform Edge (Extensibility)
- The conversion from Domain Markdown -> PDF ByteArray must happen at the absolute edge of the Android OS platform, completely outside the Prism Domain.

## 4. Anti-Hallucination & Anti-Illusion Proofing

To ensure the LLM never hallucinates an `AssetId` or lies about an artifact existing:

1. **Information Asymmetry Guarantee**: The LLM cannot invent artifacts. The `ArtifactSniffer` is a hard-coded deterministic lookup engine (L1 Native Code). The LLM's prompt is *only* injected with an Artifact Summary **if and only if** the Sniffer legitimately retrieved it from the physical SSD. 
2. **Strict Grounding**: The system prompt must instruct the LLM: *"You have access to the following artifacts: [Injected List]. Do not offer, reference, or invent any documents outside of this list."*
3. **UI Source of Truth**: The "Artifact Box" UI is driven purely by the native ViewModel, not the LLM's text output. Even if the LLM hallucinates a sentence like "Check the box", the UI box will remain empty and disabled unless the Kotlin backend physically bound a legitimate `AssetId` to the state stream.
