## Variant Role

This is a transient Harmony-native app container.

It is:

- a clean Harmony-owned root for work that cannot safely live in the Android tree
- a reduced-capability Harmony app needed now
- a Tingwu/audio-focused delivery lane

It is not:

- a full-parity Harmony port of the Android app
- the new shared product-truth owner
- the owner of the separate ArkUI page-native verification package
- permission to rewrite shared scheduler or reminder semantics

Current Android build naming for the transient split:

- full Android APK display name: `智能销售-安卓`
- Harmony-facing transient APK display name: `智能销售-鸿蒙`
- current Harmony debug bundle name for AGC/device signing: `smartsales.HOS.test`

## Supported Capability Set

The current `tingwu-container` variant supports:

- local or phone-owned audio import/selection
- Tingwu submission and progress observation
- transcript and available artifact rendering
- persisted artifact reopen without rerunning Tingwu by default
- Harmony-owned packaging, lifecycle, storage, and runtime wiring needed to deliver this reduced app honestly

Operator-only verification support inside the same Harmony root:

- bounded Harmony scheduler Path A mini lab for `Uni-A` exact create and `Uni-B` vague create
- local persistence of Harmony scheduler verification tasks and trace events
- critical-joint scheduler telemetry plus owner-chain and Path A commit visibility for backend-first development

---

## Disabled Capability Set

The current `tingwu-container` variant explicitly hides and does not support:

- scheduler create, reschedule, follow-up, and reminder surfaces
- scheduler drawer, dynamic-island scheduler lane, and scheduler discoverability teaching
- onboarding quick-start scheduler commit or shell handoff flows
- Ask-AI, audio-grounded chat, and session-binding lanes
- badge pairing, badge sync, badge download, `ConnectivityBridge`, and `BadgeAudioPipeline` hardware flows

These capability limits are Harmony-owned variant boundaries, not shared product-truth changes.

Clarification:

- the operator-only scheduler backend verification sandbox is not a user-facing scheduler surface
- it keeps only the Uni-A / Uni-B Path A mini lab visible by default
- conflict, reschedule, and null safe-fail verification may exist only in a collapsed internal scaffold section for backend completion
- it exists to prove Harmony scheduler dataflow before broad UI translation starts
- it must not be presented as scheduler parity

## Delivery Rule

This variant reuses shared Tingwu/audio intent only where the contract remains truthful:

- Tingwu stays the source-led long-form audio artifact engine
- optional provider payload gaps must degrade to transcript or other available artifacts instead of failing the whole container
- persisted artifacts may be reopened later without rerunning Tingwu by default

This variant must not inherit optional shared consumers that are out of scope here:

- no scheduler mutation lane
- no reminder delivery lane
- no chat attachment lane
- no Android-only badge hardware lane

## Current Native Runtime Slice

The current Harmony-native slice now owns a first real Tingwu container runtime inside the Harmony root.

Current live structure:

- import ingress uses Harmony document-picker flow only
- metadata persists as `harmony_tingwu_metadata.json`
- imported media persists as `harmony_<audioId>.<ext>`
- artifact payloads persist as `harmony_<audioId>_artifacts.json`
- the current ArkTS UI exposes empty state, inventory, progress, artifact detail, and the bounded scheduler mini-lab on one container page

Current config law:

- Tingwu and OSS credentials remain Harmony-owned at runtime through the generated artifact `entry/src/main/ets/config/HarmonyContainerConfig.local.ets`
- repo-root `local.properties` is only the local authoring source for developers, not the runtime source consumed by ArkTS services
- `hvigorfile.ts` is the Harmony root build-owned generator and must emit the Harmony local config artifact before compile instead of reusing Android `BuildConfig`
- tracked ArkTS code must import `HarmonyContainerRuntimeConfig.ets` rather than hard-importing the ignored generated artifact directly
- when the generated artifact is absent, the tracked runtime config entrypoint must fall back to empty config and surface missing-key state honestly instead of failing module resolution

Current implementation caution:

- OSS upload currently uses a direct Harmony HTTP PUT path that reads the selected file into memory before upload
- this is acceptable for the transient container slice, but large-file streaming hardening is still deferred
- the generated Harmony local config artifact is intentionally ignored and must never be committed
- local-properties parsing now covers escaped separators, continuation lines, and unicode escapes so Harmony local config generation stays aligned with the repo's existing Gradle source expectations
- no claim is made yet that the current ArkTS path is fully build-validated with a local Harmony toolchain in this repo session
